The two source files are: • customer_info.csv — one row per customer (customer_id, name, gender, birthdate, household composition, number of complaints, geolocation, distinct stores visited, loyalty card number, eleven lifetime category-spend variables, total distinct products purchased, year of first transaction, percentage of products bought on promotion, and typical shopping hour). • customer_basket.csv — 100,000 randomly sampled transactions, each one a (invoice_id, list_of_goods, customer_id) triple that can be joined back to customer_info through customer_id.  The deliverables required are a versioned code repository (with the GitHub user ivopbernardo invited as a collaborator), an executive report or interactive application describing the segments and campaigns, and a CSV mapping every customer_id to a final cluster. The project keeps these requirements in view throughout: customer_segments.csv and id_and_cluster.csv both contain one row per customer (33,038 rows, matching the cleaned dataset exactly), and the pipeline is split cleanly into preprocessing, modelling, characterization and campaign-design stages so that the “no code in the report” rule can be respected while still keeping the analysis reproducible. 3. Exploratory Data Analysis — Notebook 0 Notebook 0 is a pure audit of the raw customer_info file: nothing is cleaned or exported here, the goal is simply to understand what the data looks like before any decision is made. This separation of “look” from “act” is a good practice signal in itself, because it means every cleaning decision documented in Notebook 1 can be traced back to an observation made here. 3.1 Duplicate and “logical duplicate” analysis The notebook first checks for exact duplicate rows (none found) and then for “logical duplicates” — records that share the same customer_name and customer_birthdate, which could indicate the same person registered twice. It goes one step further and inspects whether repeated surnames combined with close geolocation might indicate members of the same household. The conclusion is balanced and realistic: surname repetition alone is too common to be informative (many surnames recur across thousands of unrelated people), so surname is explicitly kept out of the modelling features and treated, at most, as a weak exploratory signal.
